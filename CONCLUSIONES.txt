PROBLEMA #1: Validación de URL Demasiado Estricta
Error: expected 'https://www.demoblaze.com/index.html' to equal 
       'https://www.demoblaze.com/'

Causa: La app redirige a /index.html pero el test esperaba exactamente /

Solución:
  ANTES:  cy.url().should('eq', Cypress.config().baseUrl + '/')
  AHORA:  cy.url().should('include', Cypress.config().baseUrl)

Archivos: catalogPage.js (línea 73), cypress/e2e/e2e/purchase-flow.cy.js (líneas 45, 92)


 PROBLEMA #2: Alert Stub Pierde Contexto
Error: expected alertStub to have been called with 'Product added', 
       but it was never called

Causa: El stub de window.alert se pierde en la navegación SPA

 Solución - Validar por API en lugar de UI:
  ANTES:  
    const stub = cy.stub();
    cy.on('window:alert', stub);
    expect(stub).to.be.calledWith('Product added');

  AHORA:
    cy.intercept('POST', '**/addtocart').as('addToCart');
    cy.wait('@addToCart').its('response.statusCode').should('eq', 200);

Archivo: catalogPage.js (método addProductToCart)

===============================================================================
                            LECCIONES APRENDIDAS
===============================================================================

1. Validaciones Flexibles
   • Usar .should('include') en lugar de .should('eq') para URLs
   • Evitar validaciones demasiado estrictas que generen falsos positivos

2. API over UI
   • Preferir cy.intercept() sobre validaciones de UI (alerts, modals)
   • Validar status codes HTTP en lugar de elementos volátiles

3. Page Object Model
   • 1 cambio en catalogPage.js → 3 tests corregidos
   • Mejora mantenibilidad y reutilización

4. Debugging Visual
   • Screenshots en puntos clave aceleran el debugging

5. Reintentos para Flakiness
   • Configurar retries ayuda con sitios inestables
   • No usarlos para ocultar bugs reales
